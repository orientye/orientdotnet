# Orient / CRpc TODO

> 2026-07-19 按当前实现重新整理。已落地的历史 TODO 见文末归档；本文件只跟踪还需要继续推进的事项。

## P1：协议与并发模型完善

1. `OrientTask` 取消语义。
   - 目前 pending call 超时和连接关闭会 fault，但没有面向调用方的 cancellation token。
   - 如要支持业务取消，必须保证 cancel completion 仍只在 owner `OrientExecutor` 线程触发。

2. 跨 executor 调度与多 executor 路由。
   - 仍缺按 `serviceId`、连接或 shard 路由到多个业务 executor 的 `ExecutorRoute` / dispatcher。

3. 连接与重连策略。
   - 客户端断线会 fault pending calls，并抛 `ConnectionLost`。
   - 是否自动重连、重连期间调用如何排队/失败、服务发现变更如何切换 endpoint，还需要单独设计。

## P2：可观测性、测试和代码卫生

1. 增加 metrics / tracing 钩子。
   - 需要能观察 QPS、pending 数、超时率、连接数、push 失败、请求耗时分布。
   - tracing 可以先预留 OpenTelemetry / W3C TraceContext 入口，不必一次性接完整链路。

2. 扩充端到端与压力测试。
   - 已有 loop、timer、codec、pipeline、heartbeat、client pending、push、server handler、`RpcServiceRegistry` 等单测（`Tests/Orient.Tests`）。
   - 仍建议补真实 client + server TCP 集成测试：正常响应、未知 service 错误、超时、断线、并发、慢消费者、超过 frame 上限。
   - 减少通过反射读取私有字段的测试，优先从公开行为断言。

3. 清理命名和占位代码。
   - `CRpcContext` 已实现 `IRpcContext` 并携带连接，但 `IRpcContext` 仍为空接口。
   - Nacos 相关路径已迁至 `Orient.Rpc/Mgr/Nacos`、`Orient.Rpc/Registry/Nacos`、`Orient.Rpc/ConfigCenter/Nacos`；其余命名空间风格可继续统一。
   - `Orient.Rpc/Mgr/Nacos/Test/NacosTest.cs` 仍在生产工程中，应移到测试/示例，或删除。

4. Nacos 组件改造成可注入服务。
   - `NacosMgr.Instance` 是强单例，`Srv` 可为空且很多方法静默返回默认值。
   - `NacosRegistry` / `NacosConfig` 目前直接暴露或隐藏 `Task` API，需要明确这是外部 interop 边界，并加初始化状态检查。

5. 错误模型增强（现状可用，按需推进）。
   - 现状：框架层已有 `CRpcStatusCode` 分段（1000–1999 框架 / 10001+ 应用）与 `RpcServiceInvoker` 框架错误响应；应用返回 `(int, byte[])` 裸 int，够内部单语言场景用，暂不需要 `CRpcError` 信封 / 字段级 details。
   - 需要修：客户端 codegen 把所有非零码折叠成 `(-1, null)`（见 `OrientCrpcGen.cs` 生成的 `*Async`），丢掉了服务端已区分的 `resultCode`。应改为透传真实 code（如 `return (result, null);`），调用方才能区分 MethodNotFound / InternalError / 应用错误。纯 codegen 改动，不动协议与 wire。
   - 按需再做（出现跨语言客户端 / 对外 API / 需向调用方传 message 时）：`CRpcError` 信封（code + message + 可选 details）、异常 → 稳定框架码的集中映射；异常 detail 可写入现有服务端日志，默认不向客户端暴露 stack。

## P3：依赖与长期演进

1. 依赖版本治理。
   - `Google.Protobuf` / `Google.Protobuf.Tools` 仍是 `3.35.0-rc1`。
   - 建议加 `Directory.Packages.props` 统一版本，并优先固定到稳定版。

2. DotNetty 替换路线。
   - DotNetty `0.7.6` 长期维护风险仍在。
   - 当前已有 `TcpChannelHost`、`IChannelPipelineFactory` 等传输边界雏形；继续把协议/业务层与 DotNetty 解耦，为 System.IO.Pipelines、Kestrel transport、SuperSocket 或直接 Socket 留迁移空间。
